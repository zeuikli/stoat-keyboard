# Stoat（白鼬注音）CHANGELOG

iOS 自製注音鍵盤（自建 librime.xcframework，從 macOS Squirrel SPEC 移植）。
本檔依時序由舊到新，每個里程碑附一條 Debug／設計 Insight。
真相來源：`SPEC-iOS.md`（章節 § 為錨；早期無逐版 build 號，最近四版標實際版本）。

---

## 階段一 — 專案啟動到可安裝（M0 → M-B1，§19–§22）

- IPA 編譯成功 → 可自簽 → 補 App Icon → 實機安裝驗證 → 接上真 librime 預編譯詞庫。
- 確立「免裝機部署」：bundle 內帶預編譯 `.bin`，`prebuilt_data_dir` 指過去，避免裝置端重建。

> **Insight｜先打通最薄的端到端，再加功能。**
> 第一刀的目標不是功能多，而是「能編譯出 IPA、能自簽、能裝上機、能查到一個字」。把最薄的一條鏈路打通，後面每個功能才有可驗證的地基；反過來先堆功能，會卡在「裝不上機」這種底層問題上空轉。

---

## 階段二 — 鍵盤布局與功能對齊原廠（§23–§49）

- 大千注音布局、功能列圖示、英文 QWERTY 頁、數字列、配色/字型，逐項貼近 Apple 原廠注音。
- 關鍵洞察：**大千鍵位 ≡ QWERTY 實體鍵位**（ㄆ=q、ㄇ=a、ㄈ=z…），所以注音鍵可直接上下划輸入英文（§48）。

> **Insight｜對齊原廠＝對齊「實體鍵位」這個不變量。**
> 注音、英文、上下划三套輸入能共存，是因為它們底層共用同一組實體鍵碼。找到這個共用不變量後，一個鍵能同時服務三種用途，不必為每種模式各畫一套。

## 階段三 — 移除語音子系統（§51–§52）

- 移除設定頁「語音輸入」並清理孤兒程式碼。
- 原因：自訂鍵盤沙盒無法錄音／叫起系統聽寫，語音註定不可用。

> **Insight｜平台硬約束面前，最好的功能是「砍掉」。**
> 與其留一個永遠不會動的語音入口製造困惑，不如承認 iOS 沙盒的硬限制、整段移除。減法也是進度。

## 階段四 — 手感與跑版 Bug（§53–§65）

- Telegram 鍵盤被切／重疊／頂部灰帶、⌫ 長按不連續刪除、自備 🌐 切換鍵、App Group 套用時機等。
- 側載致命坑：**App Group 在重簽後失效** → 鍵盤改讀自己的 `UserDefaults`，並做「鍵盤內建選項選單」取代跨進程共享（§63/§65）。

> **Insight｜側載環境 ≠ 正式簽章環境，要按側載的現實設計。**
> App Group、跨進程共享這類「正常情況下會動」的機制，在重簽側載下會靜默失效。容易誤判成「程式有 bug」，其實是部署模型不同。針對實際運行環境（側載）設計，而不是針對理想環境。

## 階段五 — 功能擴充：RIME 精髓（§66–§69，含 §35–§42）

- 聲韻母亂序（free-order，「這是 RIME 的精髓」）、emoji 候選（哈哈哈→🤣）、顏文字候選、智慧候選層、生僻字 tofu 過濾。
- `algebra` 是編譯期、不可執行期切換；`simplifier@dict + option_name` 才可執行期開關（§67/§68 的分界）。

> **Insight｜分清「編譯期」與「執行期」能改的東西。**
> RIME schema 有些規則（algebra/derive）要重新 build `.bin` 才生效，有些（simplifier 開關）可即時切。把「使用者設定能切的」對應到執行期機制、「結構性規則」放編譯期，才不會做出「設定切了卻沒反應」的假開關。

## 階段七 — 選字優化 + 詞庫排序（§74–§75）

- pull 最新洋蔥詞庫；發現 octagram 文法雖載入但 `contextual_suggestions` 沒接 → 補上，候選重排序生效。

> **Insight｜「載入了」不等於「生效了」。**

## 0.1.90 — 修玻璃浮凸異常 + 移除無效「預設英文模式」（§92）

- iOS 26「玻璃按鍵」：系統 `UIButton.Configuration.glass()` 材質在實機渲染不穩（一台浮凸陰影、另一台藍底）→ 改用可預期的**半透明霜白**（換底色、沿用細陰影圓角），打開不再壞。

> **Insight｜系統渲染的東西，不同裝置會給你不同的壞法。**
> 同一段 glass 程式碼在兩台機器分別壞成藍底與浮凸——系統材質的表現吃裝置/OS 版本，本機又無 GUI 可驗。與其賭一個修不修得好的系統材質，不如換成「自己畫、可預期」的近似效果：寧可樸素但穩定，也不要華麗但隨機壞。

## 0.1.91 — 版本分層樣式 + 真 Liquid Glass 探索（§93–§97）

- **版本分層（A+B）**：新增 `isOS26` 閘 + `KeyStyle` 概念。iOS 26 套圓角/未來感視覺，iOS 16–18 走原廠 classic（方正、circular）——以「該版原廠長怎樣」分層，不以 API 可用性分層。
- **真 Liquid Glass 探索**：依官方 SDK（`UIGlassEffect` / `UIGlassContainerEffect`）做了 3 個獨立測試 IPA（per-key → 官方容器 + 巢狀 glass）。每版實機都有不同破圖（圖示鍵扭曲鏡像、切 App 拉長藥丸）。

> **Insight｜系統渲染的東西，不同裝置給你不同的壞法，且本機無 GUI 無法驗。**
> 真玻璃在密集鍵盤上，兩種官方做法都有結構性脆弱（塞按鈕內 → 圖示破圖；獨立層靠 frame 同步 → 切 App 破圖），每修一輪都是盲賭。最終擱置，主版用穩定的「霜白 glass-lite」。寧可樸素穩定，不要華麗但隨機壞。

## 0.1.94–0.1.97 — 用模擬器精確比對原廠 iOS 26（§100–§104）

- **建 KBHarness（截圖驗證 target）**：把 `KeyboardViewController` 嵌進測試 App + 加 native 模式（叫出系統鍵盤），用 simctl 截圖 + 像素取色，**量到原廠真值**：
  - 底色 淺 `#E2E4E8` / 深 `#171717`；鍵 白 / `#3D3D3D`；**功能鍵與內容鍵同色**（iOS 26 無灰功能鍵，靠 icon/位置區分）。
  - 上緣圓角 ~26pt（= 系統鍵盤外框）。
- **底色/鍵色**改為原廠精確量測值（撤掉用猜的 RGB、撤掉在非鍵盤情境渲染透明的 `UIInputView.keyboard`）。
- **上緣圓角**多輪收斂：圓角必須 = 系統外框（26pt）才「切齊」——小一點露綠縫、大一點露方角且削到注音 preedit；削字問題用「**內容讓開角落**」（preedit 左內距）解，而非縮圓角。

> **Insight｜量測 > 猜測；先找對「不變量」。**
> 顏色與圓角不是自由參數——底色是系統量得的定值，圓角必須=宿主外框。建一個能截圖取色的 harness，把「猜」變成「量」，是這段最大的槓桿；本機無法驗的（真鍵盤外框互動）才退而靠實機回報。

## 0.1.98–0.1.109 — 原廠配色定版、深色、流暢度、切 App 穩定（§105–§116）

- **系統鍵盤底材**（§105）：改用 `UIInputView(inputViewStyle:.keyboard)` 作最底層，實機渲染原廠材質、自動深淺/半透——撤掉先前用猜的底色。

- **動作 return 鍵藍/灰**（§109–§110）：依 `textDocumentProxy.returnKeyType` 判動作型（.go/.send…→藍）vs 預設型（灰）；`enablesReturnKeyAutomatically` 實測 host 不轉發 → 改近似（動作欄位空白灰、有字/組字藍）。

> **Insight｜外觀別猜、用 SDK 語意色；抖動先找「那條看不見的系統約束」。**
> 深色鍵色不是調 RGB，是對上 Apple 自適應灰家族；切 App 變形的真凶是一條系統自己加的高度約束，不是我們的版面算錯。

## 0.1.110–0.1.114 — 切換動畫對標原廠、消割裂感（§120–§125）

- **不疊加自家動畫**（§120, v0.1.110）：所有高度約束變更包進 `performWithoutAnimation`——切 App 時本方法在系統 animated layout pass 內被呼叫，直接改高會被外層動畫接管成彈跳；原廠鍵盤也不動畫 height。
- **第一列標點/顏文字改 ⚙ 可開關**（§121, v0.1.111）：idle 第一列快捷列拆「標點」「顏文字」兩段，⚙ 選單各自開關（預設開）。
- **兩段全關→收掉空候選列**（§123, v0.1.112）：兩段都關且 idle 時，連空候選列一併隱藏（不留空白帶）、鍵盤對應變矮；保留 ⚙ 以便再開回來；打字出候選時自動回復。
- **App resume 後乾淨重套**（§124, v0.1.113）：深入 iOS 26.5 SDK 研究（Apple DTS thread 799003）確認 app suspend→resume 時 `viewWillAppear/viewIsAppearing` 皆不 fire → 掛 `willEnterForegroundNotification` 在轉場後非動畫重套高度。
- **轉場交給系統、不在轉場期重繪**（§125, v0.1.114）：移除 `viewWillLayoutSubviews`/`viewDidAppear` 每 pass 的 `applyHeight`（高度約束持續存在、h 沒變不需重套）；轉場期只保留必要的 idempotent relax，純讓系統 snapshot/轉場動畫處理。

> **Insight｜可控的做到底，不可控的誠實說。**
> Apple DTS 證實：自訂鍵盤跨進程、只能在主 view 首繪後改高 → 首次呈現的 flash **原生無法消除**；App 切換的卡片縮放是系統動畫、原廠亦同、無 API 可改。我方能控制的只有「不疊加自家動畫 + resume 後乾淨重套 + 轉場期不重繪」——這些做到底，殘留偶發來自上述系統硬限制。

## 0.1.117 — 修「爲→為」+ 擴充詞庫（§127–§128）

- **候選正字「爲」改通行「為」**：根因是 octagram 語法 `zh-hant-t-essay-bgw` 的語料用正字爲。改用 `bgc` 語料（通行為）→ 打 ㄨㄟ 首選回到「為」，且**保留 octagram 上下文重排**（越打越準不犧牲）。
- **擴充詞庫**：dict 加 import `phrases.chtp`（81 萬詞組），對齊 Squirrel onionplus 完整度；專業/罕用詞不再需逐字組。
- **記憶體驗證**：librime 以 read-only mmap 載入 table.bin（27MB），vmmap 實測僅 ~320KB resident / 0 dirty → 不計入 keyboard extension 的 jetsam footprint，27MB 詞庫不會撐爆 ~48–66MB 上限。
- 修 prebuild bug：`ONION_GRAMMAR` 未 export 致 grammar 載入失敗（schema 指 bgw 但僅 bgc.gram 存在）→ 加 export 修正、schema 與 gram 檔一致。

> **Insight｜「字不對」未必是字庫問題——先分清詞庫權重 vs 語法重排。**
> 純注音 dict 本來就是「為」，是後掛的 octagram 語法（正字語料）把「爲」重排上來。換對語料即解，不必動字庫；而擴充詞庫靠 mmap 不吃記憶體。

## 0.1.119 — 簡化：候選列固定（§130）

- §123/§129 的「收起空第一列 + 可關」太複雜。改為**候選列永遠固定**（注音模式恆顯、高度不變、不收起），只留「第一列標點」「第一列顏文字」兩開關決定顯示內容。

## 貫穿全程的元 Insight

1. **先打通最薄的端到端，再加功能。** 地基能驗證，功能才有意義。
2. **針對實際運行環境設計**（側載/沙盒），不是理想環境。
3. **視覺類行為只有實機算數**；本機驗 build + engine，外觀標未驗、靠截圖迭代。
4. **卡住多輪 → 先懷疑需求互斥**，把矛盾攤開讓使用者選，而非默默挑一個。
5. **找對不變量**（實體鍵位、固定鍵高、啟用旗標），複雜問題會塌縮成簡單規則。
6. **退版常救不了**，根因往前修；**「載入了」≠「生效了」**，查資料流末端的開關。
